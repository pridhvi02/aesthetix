Git Commands for Effective Version Control

This page introduces some fundamental Git commands used on GitHub to manage your code effectively. Remember, you'll be using these commands in your terminal or command prompt.

1. `git init`

Initializing a Git Repository

The `git init` command is the first step. It creates a new Git repository within your local project directory. This tells Git to start tracking changes to your code files.

git init

2. `git add`

Adding Files to the Staging Area

The `git add` command tells Git which files you want to include in your next commit. You can add individual files or use wildcards like `git add *` to add all modified files in the current directory.

git add filename.txt
git add * (Add all modified files)

3. `git commit`

Committing Changes to the Repository

The `git commit` command creates a snapshot of your currently staged changes (files added with `git add`). It's crucial to provide a descriptive commit message summarizing the changes you made.

git commit -m "Added new feature X"

4. `git push`

Pushing Committed Changes to GitHub

The `git push` command uploads your local commits to a remote Git repository, typically hosted on GitHub. You'll need to have your GitHub repository set up and linked to your local project for this to work.

git push origin master (assuming your remote branch is named 'master')

Additional Notes

* You can use `git status` to check the status of your working directory, staged files, and uncommitted changes. * Consider using `git branch` to manage different development branches within your project. * Refer to the official Git documentation for more advanced commands and functionalities: https://www.git-scm.com/ By mastering these foundational Git commands, you can effectively manage your code versions and collaborate seamlessly on projects using GitHub. This page explains the purpose of each command, provides code examples, and includes additional notes for further exploration. Remember to replace `filename.txt` with the actual name of your file and adjust the remote branch name (`master` in the example) if needed.
